**Companion to:** Autonomous Build Playbook v2.1 and Master Engineering Build Spec v1.0
**Purpose:** Give the founder (you, non-developer) five capabilities the v2.1 fleet did not have:

1. A **live visual Build Console** at `console.getbankable.io` that shows everything the fleet is doing, in real time, with screenshots and demo logins.
2. A **demo environment per module and per pull request** so you can click into the work and try it like a real customer.
3. A **Decision Engine** that auto-explains the WHY behind every architectural and design choice in plain English.
4. A **Legacy Codebase Migration Protocol** (the Archeologist agent) that ingests your existing software, classifies it, and produces a precise plan for what to keep, adapt, rebuild, or deprecate.
5. A **multi-layer Backup and Disaster Recovery posture** so nothing is ever lost.

This document also adds five new agents to the fleet, bringing the total from 22 to 27.

I am writing this as the master developer and designer on your build. Decisions in this document are made, not deferred. Where I am picking the path, I name the choice and the rationale.

## AUDIT REPORT (v2.2)

Same discipline as v2.1. Here is what I verified, what I am choosing, and what limitations remain.

### Verified

- Next.js App Router is the right runtime for the Build Console; it deploys to Vercel free tier even at high traffic.
- GitHub's GraphQL API exposes everything needed for the dashboard (issues, PRs, deploys, check runs, comments).
- Vercel's Deployment Webhook and Build Output API can stream live status into the Console.
- Playwright supports automated screenshot capture per route, headless, in CI.
- Supabase has built-in Point-In-Time Recovery (PITR) on Pro plan and above.
- GitHub repos can mirror to GitLab via Settings then Mirroring repositories, fully automated.
- AWS S3 supports Object Versioning, Object Lock (compliance mode), and Cross-Region Replication, all required for the vault backup tier.
- The Anthropic Console allows creating organization-level usage caps in addition to per-key caps.
- ADR (Architecture Decision Record) is a real, well-established pattern (Michael Nygard's template).
- Magic-link demo authentication is supported natively by Supabase Auth.

### Decisions I made on your behalf (master-developer hat)

1. **Build Console URL.** `console.getbankable.io`. Subdomain on the same Cloudflare zone. Public, read-only, password-gated for demo-credential reveal.
2. **Console framework.** Same Next.js app as the product (`apps/web`) but a separate route group `(console)` so it shares design tokens and components. Avoids a second deploy target.
3. **Milestone model (replaces sprints).** Work is grouped by logical milestone, not by calendar. A milestone = a cluster of issues whose closure delivers a coherent product capability. Examples: `M1 Auth Online`, `M2 Identity & Tenants`, `M3 Diagnostic v0`, `M4 Vault v0`. Milestones complete when their issue cluster closes. There is no week or day attached. This gives you the visual progress feel of a sprint without the time constraint.
4. **Demo identity model.** One pre-seeded demo tenant per preview deploy, with one demo user per role (`demo-client@`, `demo-advisor@`, `demo-admin@`, `demo-compliance@`, `demo-lender@`). Magic-link links are generated by a Console endpoint and shown in the dashboard. Each preview ephemeral tenant is destroyed on PR close.
5. **Why-Doc model.** Every ADR gets a `plain-english.md` sibling generated by `bot-explainer`. The Console renders both side by side, plain English on top, technical detail expandable below.
6. **Legacy review entry point.** You give the Archeologist agent a GitHub URL, a local path, or a ZIP upload. It runs a classification pass and outputs a Migration Report PR. You approve, the Orchestrator converts findings into issues.
7. **Backup strategy.** Five layers, defined in the Backup section. The fleet itself owns operating Layers 2 through 5. Layer 1 is git, which is automatic.

### Limitations that remain

- **Console can show but not control.** The Console is read-only. Operator actions like `compliance-officer-approved` and `promote-to-prod` still happen as GitHub comments. This is deliberate. A control plane that can ship to production must be authenticated and audited; comments already are.
- **Demo data is synthetic.** Demo tenants will never contain real PII, real credit, or real bank data. The fleet cannot show you "what a real customer would see" until you have real customers.
- **Archeologist can read but not assume.** It will classify your existing code, but it cannot know your business context until you tell it. The Archeologist prompt below explicitly asks you for context up front.
- **Layer 4 backup (offline tarball) requires a human.** Plug in the external drive once a quarter. Everything else is automated.

## PART B. THE DECISION ENGINE (THE WHY)

### What it is

Every meaningful architectural, infra, security, compliance, AI, and UX decision the fleet makes is recorded as an ADR. The Decision Engine adds two things:

1. An **upgraded ADR template** that forces structured reasoning.
2. A **plain-English translator agent** (`bot-explainer`) that generates a non-technical version of each ADR.

You read the plain English. You drill into the technical detail only if you want.

### The upgraded ADR template (saved at `docs/adr/_template.md`)

```markdown
# ADR-{{N}}: {{Title}}

**Status:** Proposed | Accepted | Superseded by ADR-{{N+x}}
**Author agent:** bot-architect
**Date:** {{ISO date}}
**Owner label:** architect | service:bai | ...

## The question
What decision were we forced to make? One sentence.

## The choice we made
What we picked. One paragraph.

## Why we picked it (the WHY)
The forcing function. The constraint. The capability we needed. Three sentences max.

## Alternatives we considered and why we did not pick them
- Option A: why not.
- Option B: why not.
- Option C: why not.

## What we give up by picking this
The honest tradeoff. The thing the other options would have given us that we now do not get.

## What we get that the others would not have given us
The specific upside.

## Consequences and follow-ups
What this forces other decisions to look like. What new constraints this creates.

## Plain English summary (generated by bot-explainer)
[Auto-filled. Three short paragraphs. Zero jargon. Reading level: a smart non-developer.]
```

### The translator agent (bot-explainer)

New agent. Owner label `decision`. Watches changes in `docs/adr/`. On every ADR commit, it:

1. Reads the ADR.
2. Generates a plain-English version targeting reading level 9 (smart adult, no jargon).
3. Writes it into the `## Plain English summary` block.
4. Commits the change with `docs(adr): plain-english for ADR-{{N}}`.

The Console renders both views, plain English on top.

### Examples of decisions this captures

- Why Next.js App Router instead of Pages Router.
- Why Supabase for MVP instead of raw RDS + Auth0 + S3.
- Why Drizzle ORM instead of Prisma.
- Why XGBoost for BPFS v1 instead of a neural net.
- Why Pinecone instead of Weaviate.
- Why Temporal in Phase 2 instead of MVP.
- Why one GitHub App for bots instead of 22 user accounts.
- Why Anthropic Claude as the LLM and which model.
- Why FastAPI instead of Django.
- Why Stripe Connect for advisor payouts instead of manual ACH.

Every choice on the spec is one of these. The Decision Engine surfaces them all without you ever opening a code file.

## PART D. LEGACY CODEBASE MIGRATION (THE ARCHEOLOGIST)

### What it is

You already have software (Fundabl, HUSLConnect, BankableIQ early code, NXTWave portals, Roccstar Wireless, Capital Connection Community). You do not throw it away. You do not assume it all carries over. You audit it. You classify every file. You decide what becomes part of the new platform.

The Archeologist agent does this work. You point it at a repo (or ZIP). It produces a Migration Report and a set of issues the fleet then executes.

### The classification system

Every file, module, integration, schema, and screen in your existing software is classified as one of:

| Tag | Meaning | What happens next |
|-----|---------|-------------------|
| **KEEP** | Already aligned with the new spec. Drop into the new monorepo with no change. | Archeologist opens a `chore: import {{path}}` PR. |
| **ADAPT** | Useful, but needs to fit new patterns (types, RLS, multi-tenant, compliance gate). | Archeologist opens a `feat: adapt {{path}}` issue with a delta plan. |
| **REWRITE** | Functionality is needed, code is not salvageable (wrong language, wrong architecture, security issues). | Archeologist closes the legacy file out, opens a spec-aligned issue under the matching owner agent. |
| **DEPRECATE** | The capability is no longer in the new spec, or it is in a sibling platform (HUSLConnect, HUSLFi). | Archeologist documents the deprecation, recommends a sunset path. |
| **MIGRATE-DATA** | Schema or data only. Build a one-time importer. | Archeologist opens a `chore: import data from {{source}}` issue under bot-platform. |

### The exact steps to get Claude Code to review your existing software

This is the step-by-step you asked for.

**Step 1. Decide which existing repos to audit.**

Tell the fleet which ones. Examples:

- `github.com/youruser/fundabl-app`
- `github.com/youruser/huslconnect-crm`
- `github.com/youruser/bankable-prototype`
- A local folder `/Users/kevin/projects/bankable-v0`
- A ZIP file uploaded into Supabase Storage

**Step 2. Grant access.**

For GitHub: add the `BANKABLE Fleet` GitHub App as a collaborator on each legacy repo with read-only access. The Archeologist needs only `contents:read` and `metadata:read`.

For local folder or ZIP: upload to a Supabase Storage bucket named `legacy-intake/` that the Archeologist polls.

**Step 3. Open the intake issue.**

Open one issue per legacy source. Title: `archeology: review {{source}}`. Label: `archeologist`. Body uses this template:

```yaml
---
owner: archeologist
source_type: github | local | zip
source_url: https://github.com/youruser/fundabl-app
context: |
  Fundabl is our legacy lending intake form built on Webflow + Airtable + Zapier.
  Real production data. 14,000 historical applicants.
  Current owner: Kevin. Current users: ~600/mo.
  Why we are auditing: deciding what to migrate into BANKABLE IQ Module 1 + 4.
known_strengths:
  - Solid acquisition funnel
  - 2 years of clean intake form data
known_weaknesses:
  - No multi-tenant
  - No real auth
  - Compliance posture is informal
classification_targets:
  - schema mapping to BANKABLE IQ schemas
  - integration mapping to BANKABLE IQ services
  - UI mapping to BANKABLE IQ screens
  - data migration mapping
---
```

**Step 4. Run the Archeologist.**

Comment `/run` on the intake issue. `bot-archeologist` spawns.

The Archeologist does this work, in order:

1. Clones / reads the source.
2. Builds a file inventory with sizes, languages, last-modified dates.
3. Maps every file to one of the 35 spec schemas, 14 spec services, or 8 spec UI surfaces.
4. Runs a security scan (Semgrep) on every file to surface red flags.
5. Reads READMEs and inline comments to understand intent.
6. For each file or module, picks one of the five classification tags.
7. Estimates effort to ADAPT vs REWRITE for every non-KEEP item, in tokens and time.
8. Generates the Migration Report.

**Step 5. The Migration Report.**

Written to `docs/migration/{{source-slug}}-report.md` and posted as a comment on the intake issue. Structure:

```markdown
# Migration Report: {{source}}

## Executive summary (plain English)
What this code does, how it fits, what we keep, what we change.

## Inventory
- Total files: N
- Lines of code: N
- Primary language: ...
- Last meaningful commit: ...
- Security flags: N high, N medium, N low

## File-by-file classification (table)
| Path | Spec area | Tag | Effort | Notes |
| src/forms/intake.tsx | Module 1 Diagnostic | ADAPT | 6h | Reuse field structure, rewire to new RLS + Zod |
| ...

## Schema mapping
| Legacy table | New schema.table | Migration approach |

## Integration mapping
| Legacy integration | New service | Replace / port / drop |

## UI mapping
| Legacy screen | New screen | Reuse / redesign |

## Data migration plan
Step by step.

## Recommended order of work
1. Schema migration (must be first, blocks everything else)
2. Auth migration (RLS + tenant)
3. UI ports
4. Integration ports
5. Sunset of legacy
```

**Step 6. You approve.**

You read the plain-English summary. You either:

- Comment `migration-approved` on the intake issue to convert findings into fleet issues.
- Comment specific overrides like `override: src/forms/intake.tsx KEEP` if you disagree with a classification.

**Step 7. The fleet executes.**

The Orchestrator reads the approved Migration Report. For each ADAPT / REWRITE / MIGRATE-DATA finding, it opens a normal fleet issue labeled with the correct owner (bot-vault, bot-diagnostic, bot-platform, etc.). The relevant agent picks it up and does the work. The legacy source code remains untouched. New code lives in the new monorepo.

**Step 8. Legacy sunset (when you say go).**

When the Migration Report is fully executed and the new platform is running the same capabilities, the Archeologist opens a final `chore: sunset {{source}}` PR that archives the legacy repo (read-only mode in GitHub) and redirects any DNS or webhook endpoints to the new platform.

### The Archeologist agent (bot-archeologist)

New agent. Owner label `archeologist`. Single-purpose: ingest, classify, plan, hand off. Does not write product code itself. The fleet handles execution.

## PART E. BACKUP AND DISASTER RECOVERY (NOTHING IS EVER LOST)

Five layers. The fleet operates four of them. You operate one (Layer 4, once a quarter).

### Layer 1. Git (automatic, no action required)

Every commit lives in the local clone, the local worktrees, and the GitHub remote. Three copies minimum, every time.

### Layer 2. GitHub to GitLab mirror (automated, set up once)

Create a free GitLab account. In GitHub repo Settings then Mirroring Repositories, configure a push mirror to GitLab. Every push to GitHub propagates to GitLab within minutes. If GitHub goes down or your account is locked, the repo lives intact on GitLab. Same protection for documents, ADRs, runbooks, prompts, everything.

The Platform agent owns the runbook `docs/runbooks/restore-from-gitlab.md`.

### Layer 3. Daily encrypted backup vault (automated, owned by bot-backup)

A separate AWS account named `bankable-backup-vault` (best practice: never store backups in the same AWS account as the primary). Daily automation:

- Postgres: `pg_dump` of Supabase database, compressed, encrypted with KMS, written to `s3://bankable-backup-vault/db/{{date}}.dump.enc`.
- Vault S3 bucket: Cross-Region Replication from `us-east-1` to `us-west-2` in the backup account. Versioned. Object Lock for tax and contract documents (compliance mode).
- Code: nightly `git bundle` of the monorepo written to `s3://bankable-backup-vault/code/{{date}}.bundle`.
- Configurations: Terraform/CDK state files snapshotted nightly.
- Compliance audit log: hash-chained log exported nightly to `s3://bankable-backup-vault/audit/{{date}}.jsonl.enc`.

Retention: 30 daily, 12 monthly, 7 yearly. Lifecycle policies move older snapshots to Glacier Deep Archive.

### Layer 4. Quarterly offline backup (manual, you do this once a quarter)

Once a quarter the Backup agent opens an issue `backup: quarterly offline drill`. You plug in your external encrypted drive. Run:

```bash
pnpm backup:offline --target /Volumes/BANKABLE-COLD-2026Q2
```

This dumps the same Layer 3 set to your drive, verifies hashes, and prints a signed receipt. You unplug, store the drive somewhere not your office. Close the issue.

The Backup agent then opens the next quarter's issue 89 days out so it pings you before the cadence slips.

### Layer 5. Disaster recovery drills (automated, quarterly)

The Backup agent runs a quarterly recovery drill:

- Spin up a clean Supabase project in a staging org.
- Restore the latest Layer 3 dump.
- Boot a fresh Vercel preview pointing at the restored DB.
- Walk through a Playwright e2e that proves the platform comes back to life with the demo seed.
- Report results to the Console Backup panel.

Pass or fail, you see it on the dashboard. If it fails, an `incident:` issue opens and Observability triages.

### The Backup agent (bot-backup)

New agent. Owner label `backup`. Owns the AWS backup account configuration, the lifecycle policies, the encryption keys, the nightly jobs, the quarterly drills, the runbooks at `docs/runbooks/disaster-recovery.md` and `docs/runbooks/restore-from-gitlab.md`.
