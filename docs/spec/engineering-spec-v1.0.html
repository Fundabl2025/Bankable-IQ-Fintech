§ 1.2 · EXECUTIVE TECHNICAL SUMMARY

Executive Technical Summary

BANKABLE IQ™ is an AI-enabled, multi-tenant, multi-sided B2B2C platform that operationalizes the Institutional Readiness Intelligence Framework™ for three audiences simultaneously: business owner clients, Certified Capital Advisors (CCAs), and institutional lender partners. This document is the master engineering build specification — the document an engineering hire or contracted dev shop reads to scope, architect, and build the production platform.

What the Platform Does (Engineering View)

  • Captures business and personal data via Plaid, tri-bureau APIs, business credit bureaus, accounting integrations, and direct upload
  • Scores institutional readiness across 7 Cs and 20 Focus Areas (Bankability Score 0-100) and credit position (Credit Position Score 0-100)
  • Adapts recommendations per business, industry, geography, and current market via the Adaptive Intelligence Engine (BAI)
  • Matches clients to capital products with per-lender predicted approval probability (BPFS)
  • Packages lender-ready submissions via AI-powered packaging engine (replaces third-party back-office)
  • Coordinates a credentialed advisor network (CCA / CCRA / CIRS / Infrastructure Partner) running on the platform
  • Licenses deal flow to institutional capital providers via the Capital Marketplace API
  • Distributes via EDO B2B licensing to SBDCs, CDFIs, chambers, franchise systems, and accelerators

Design Principles

PrincipleRationale
Modular monolith first, microservices only when neededStart with Node.js modular monolith for application services, Python microservice for AI. Decompose to microservices only when scale demands it. Avoids premature distributed-systems complexity.
Event-driven for cross-module communicationModules communicate via events (Plaid received → Score Engine subscribes → recomputes Bankability Score → emits ScoreUpdated event). Loose coupling, easy to evolve.
Multi-tenant from day 1Every row in every schema has tenant_id. Row-level security (RLS) in PostgreSQL enforces isolation. EDO licensees get their own logical tenant.
Append-only event log for audit + complianceEvery state change writes to an immutable event log (Kafka topic + S3 archive). Critical for CROA/FCRA/ECOA audit requirements and for the Telemetry Layer.
Background jobs for slow operationsCredit bureau pulls, lender packaging, AI generation all run as background jobs (Temporal). UI shows progress, never blocks.
Read replicas for analyticsHeavy analytics queries (CCA portfolio dashboards, HQ metrics, investor reports) hit a read replica or Snowflake, never the primary.
Feature flags for safe rolloutEvery new feature behind a flag (LaunchDarkly or self-hosted). Gradual rollout: 1% → 10% → 50% → 100%. Instant rollback.

2.3.1 Inter-Service Contract Rules

  • No shared databases. Each service owns its schema. Cross-service reads happen over the API.
  • Synchronous calls are REST (HTTP/2 + JSON) when latency-critical, gRPC for high-throughput service-to-service.
  • Asynchronous events publish to the event bus (NATS JetStream in dev, AWS EventBridge + SQS in prod) with at-least-once delivery.
  • Schema evolution follows additive-only rules for events; breaking changes require a new event version and a transition window.
  • Idempotency keys are required on every state-mutating API endpoint.

3.1 Module 1 — Diagnostic Engine

Purpose: Score a business across 30 readiness domains; produce per-domain factor breakdowns, gaps, and prioritized fixes.

3.1.1 Inputs

  • Self-reported intake (10 sections, ~140 fields, branching logic)
  • Vault-extracted documents (formation docs, EIN letter, financials, tax returns, bank statements)
  • Connected accounts (Plaid bank/cash flow, accounting via QuickBooks/Xero, IRS transcripts)
  • Bureau data from credit-svc

3.1.2 Outputs

  • Overall Readiness Index (0–1000)
  • Per-domain score (0–100) with factor weights
  • Ranked gap list (top N gaps by Capital-Impact-Score)
  • Source-of-truth lineage for every factor (auditable for FCRA)

3.1.3 Internal Components

ComponentResponsibilityTech
Intake OrchestratorForm state machine, branching, validationXState + React Hook Form
Factor CalculatorDeterministic rules engine for 800+ factorsPython + JSON-defined rules
Score AggregatorWeighted roll-up to domain + overall scoresNumPy / pandas
Gap RankerPrioritize fixes by capital impactBPFS impact deltas
Snapshot WriterImmutable scored snapshots for auditPostgreSQL + S3

3.2 Module 2 — Compliance Engine

Purpose: Enforce CROA, FCRA, ECOA, GLBA, and state-level credit-services laws as a non-bypassable layer in every flow.

3.2.1 Enforcement Surface

  • Pre-action gates: every credit pull, marketing message, lender match, and document share routes through Compliance Engine first.
  • Disclosure rendering: contextual disclosures injected by component into UI; rendered version + timestamp captured.
  • Adverse action: when a lender declines, ECOA-compliant notice is generated automatically.
  • Right to dispute: bureau-aware dispute flow with letter generation and tracking.
  • Immutable audit log: append-only, hash-chained, retained for the longer of 7 years or regulatory minimum.

3.2.2 Rule Storage

Rules live in versioned YAML, signed by Compliance Officer on commit. Hot-reload via signed config push. No code deploy required for rule changes.

3.3 Module 3 — AI Coaching System

Purpose: Deliver context-aware coaching to clients and advisors using Anthropic Claude with retrieval over the client’s own data plus the Bankable Knowledge Graph.

3.3.1 Prompt Architecture

System Prompt (locked, signed by Compliance)
+ Client Context Pack (RAG: top-k from client vault + scores)
+ Advisor Notes (if advisor-facing)
+ Compliance Guardrails (forbidden topics, required disclaimers)
+ User Turn
→ Claude (with structured-output schema)
→ Validator (schema + compliance scan)
→ Render

3.3.2 Retrieval Index

  • Pinecone (or Weaviate) per-tenant namespace
  • Embedding model: Claude embeddings (or OpenAI text-embedding-3-large as fallback)
  • Chunk policy: 512-token chunks with 64-token overlap, document-aware splitting
  • Refresh: event-driven on vault updates, nightly full rebuild

3.5 Module 5 — Document Vault

Purpose: Encrypted, classified, retention-aware document store with OCR, extraction, and tamper-evident audit trail.

3.5.1 Storage

  • S3 with SSE-KMS, per-tenant CMK (Customer Master Key)
  • Object lock in compliance mode for tax returns and signed contracts
  • Presigned URLs only, max 5-minute TTL, IP-restricted when possible

3.5.2 Processing Pipeline

Upload → Antivirus (ClamAV) → MIME check → KMS encrypt
       → Textract (OCR) → Layout parser → Field extractor
       → Classifier (doc type) → Index (Pinecone) → Notify pathway-svc

3.6 Module 6 — Pathway Engine

Purpose: Orchestrate the Goal-Backwards Build-Out — given a target funding outcome, generate and run the multi-step plan.

3.6.1 Runtime

Temporal workflows. Every pathway is a long-running, durable workflow with retries, compensation, and human-in-the-loop signals.

3.6.2 Pathway Schema

Pathway
├── Goal (target outcome: Capital Today | Bankability Built | Institutional Access)
├── Target Amount + Timeline
├── Current State Snapshot (from diagnostic)
├── Required State (computed from goal)
├── Gap Set (ordered)
├── Steps[] (each = action, owner, due, blockers, dependencies)
└── Checkpoints[] (auto-reevaluate every N days)

5.1 API Style and Conventions

  • External clients (web, mobile, partners): REST + JSON over HTTPS, versioned with /v1/ path prefix
  • Service-to-service: gRPC with Protocol Buffers for high-throughput paths; REST for everything else
  • Authentication: Bearer JWT from Auth0; service-to-service uses short-lived mTLS certs
  • Idempotency: Idempotency-Key header required on POST/PATCH/DELETE
  • Pagination: cursor-based, max page size 100
  • Errors: RFC 7807 problem+json with stable type URIs
  • Rate limiting: token bucket per API key + per user; headers expose remaining quota
  • Schemas: published as OpenAPI 3.1; clients generated from spec

5.3 Webhooks (Inbound + Outbound)

6.1 Client Journey (12 Phases)

  1. Acquisition — Lands from advisor referral, ad, or marketplace search. Captured with referrer attribution.
  2. Signup & Verification — Auth0 signup, email verify, MFA prompt, ToS + privacy acceptance recorded in compliance audit log.
  3. Business Profile — EIN, legal name, NAICS, formation date, state, ownership.
  4. Diagnostic Intake — 10-section guided intake with branching; save-and-resume; document upload prompts inline.
  5. Initial Snapshot — First Readiness Index computed; Capital Outcomes preview rendered.
  6. Pathway Selection — Client picks target outcome (Capital Today / Bankability Built / Institutional Access).
  7. Pathway Activation — Goal-Backwards plan generated; first 3 steps surfaced; first checkpoint scheduled.
  8. Coaching Engagement — Daily/weekly nudges via in-app + email; conversational coach available 24/7.
  9. Document & Data Build-Out — Vault completed across required doc set for chosen outcome.
  10. Re-Diagnostic — Snapshot regenerated; BPFS updated; matches recomputed.
  11. Marketplace Match & Apply — Curated lender shortlist; one-click apply with prefilled package; consent recorded.
  12. Funding Outcome + Loop — Approval/decline captured; if declined, adverse-action notice + remediation plan triggered. If approved, success fed back to BLIN.

6.2 Advisor Journey (10 Phases)

  1. Invite & Onboarding — Receives Partner invite; completes advisor verification.
  2. White-Label Setup — Brand assets, domain (optional), email-from, payout details.
  3. Caseload Import — CSV import or per-client invite link; clients land into advisor's workspace.
  4. Daily Queue — System-ranked next-best-action list for the advisor's full book.
  5. Coaching Delivery — Send nudges, schedule reviews, drop notes, send doc requests.
  6. Progress Review — Per-client dashboard with score trajectory, pathway state, BPFS delta.
  7. Pre-Submission Audit — Compliance Engine checks every applicant package before marketplace submission.
  8. Submission & Tracking — Submit to lenders; track decisions; manage counter-offers.
  9. Revenue Settlement — Earned fees + funded-deal splits reconciled; payouts via Stripe Connect.
  10. Performance & Insight — Cohort-level analytics on funding-success, time-to-funding, drop-off.

6.3 Lender Journey (6 Phases)

  1. Partnership — Lender signs Marketplace Partner Agreement; receives partner credentials.
  2. Criteria Definition — Lender encodes credit box (FICO bands, time-in-business, revenue, NAICS).
  3. Cohort Subscription — Lender subscribes to matching cohorts; receives daily/weekly digests.
  4. Decisioning — Lender reviews opt-in applicants; submits decisions via Partner API.
  5. Funding — Lender funds approved deals directly with the client (Bankable does not touch funds).
  6. Outcome Feedback — Lender reports funded amount, terms, eventual performance; feeds BLIN learning loop.

12.1 Bankable Predictive Funding Score (BPFS) Model

  • Targets: probability of approval, expected APR, expected loan amount, expected term, conditional on a given product type
  • Algorithm: gradient-boosted trees (XGBoost) for v1; calibrated with isotonic regression; transition to a structured neural approach for v2 once data scale supports it
  • Features: ~220 features across credit, cash flow, doc completeness, time-in-business, NAICS risk, owner profile, prior funding history
  • Training cadence: weekly retrain; champion-challenger gate; production rollout via canary
  • Explainability: SHAP values surfaced per prediction; top drivers shown to advisor & client
  • Fairness: ECOA-aligned bias audit (4/5ths rule + statistical parity) every quarter on protected proxies; failing audits block promotion

12.2 BLIN — Lender Intelligence Network

  • Stores per-lender per-product success rates segmented by score-band, NAICS, region, time-in-business, revenue band
  • Updates from marketplace.application.updated events; smoothed with Bayesian priors to avoid overfitting on sparse cells
  • Serves as a re-ranking layer on top of BPFS for marketplace match

12.3 BII — Industry Intelligence Pipeline

  • Sources: Census BDS, BLS QCEW, IRS SOI, Fed Small Business Credit Survey, commercial data partners
  • NAICS rollups (2-, 3-, 4-, 6-digit) for benchmarks: typical revenue, margins, default rate, growth rate
  • dbt models in Snowflake; daily refresh; served via cached read API

12.4 BMS — Macro Signals Pipeline

  • Sources: FRED (Fed rates, yield curve), SBA loan volume, Senior Loan Officer Survey, commercial credit conditions indices
  • Stored as time-series in TimescaleDB; published as features to BPFS
  • Triggers Adaptive Wheel re-evaluation on material regime shifts

12.5 Coaching LLM Stack

  • Primary model: Anthropic Claude (latest production model)
  • Structured output via tool-use schemas for non-conversational tasks
  • Streaming SSE for chat surfaces
  • Guardrails: pre-prompt compliance system, output classifier for prohibited categories, citation-required mode for factual claims
  • Evaluation: golden-set regression run on every prompt change; human-in-the-loop sampling on 1% of completions

12.6 Bias Audit & Model Governance

  • Model registry: MLflow; every model has owner, training data lineage, evaluation report, and approval signature
  • Disparate impact tested across geography, NAICS, business age, and revenue bands as proxies
  • Decisions retain SHAP explanations for 7 years
  • Annual external model risk review
